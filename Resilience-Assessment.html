üí° What Is Zscaler Resilience?

Zscaler Resilience is a complete set of capabilities that ensures uninterrupted business continuity. It handles three categories of failure ‚Äî blackouts, brownouts, and catastrophic events ‚Äî using a combination of automatic failover, dynamic path selection, and customer-controlled disaster recovery. All capabilities are included in Business Edition and above at no additional cost, including one pair of Private Service Edges.

Understanding Each Failure Type
‚ö°
Blackout ‚Äî Data Center Outage
A Zscaler data center or upstream ISP goes completely offline
Automatic
What happens: Zscaler Client Connector detects the primary gateway is unreachable and automatically fails over to the secondary gateway. Traffic routes through the next-closest data center.
  • ZCC automatic gateway failover (built-in, no config needed)
  • Carrier-neutral data centers with multiple ISP paths
  • Over-provisioned capacity absorbs transient load spikes
  • BGP controls reroute traffic to accommodate contingencies
  • CloudOps can switch upstream ISPs while resolving primary issues
üåä
Brownout ‚Äî Performance Degradation
Service is "up" but slow ‚Äî the hardest problem to detect and resolve
Config Required
What happens: An upstream ISP or peering issue causes latency spikes. Everything technically works but users experience slowness. Without dynamic selection, traffic stays on the degraded path.
  • Dynamic Performance-based Service Edge Selection: ZCC continuously probes gateways for HTTP latency and autonomously switches to the lowest-latency path
  • Customer-controlled Data Center Exclusion: Manually set a temporary exclusion period for a data center experiencing issues ‚Äî services auto-restore when resolved
  • ZDX monitoring provides end-to-end visibility into user experience to detect brownouts in real time
üî•
Catastrophic ‚Äî Cloud Unreachable
Natural disaster, nation-state attack, or widespread infrastructure failure
Pre-config Required
What happens: The Zscaler cloud is not accessible. This is the black swan event ‚Äî extremely rare but potentially devastating without a plan.
ZIA Disaster Recovery
ZCC switches to local content filtering mode. Direct internet access restricted to critical business apps only via a hosted DNS allow list and custom PAC file. Basic URL filtering continues locally.
ZPA Disaster Recovery
Traffic redirects to ZPA Private Service Edges in your data center or cloud. Most recent policies still enforced. Private app access continues with full security controls.
  • DR mode activated via DNS records ‚Äî customer-controlled trigger
  • SAML assertion validity extended 14 days during DR
  • Upon cloud restoration, system returns to normal operation automatically
  • Critical: PSEs, DNS records, and PAC files must be configured BEFORE an event

Resilience Deployment Plan

A phased approach to building full resilience posture ‚Äî from automatic protections to catastrophic DR.

Week 1 ‚Äî Validate Existing Protections
Confirm Automatic Failover & Update ZCC
  • Verify ZCC version across all endpoints (Windows 4.0+, macOS 3.7.1+)
  • Confirm primary + secondary gateway configuration in forwarding profiles
  • Validate Dynamic Service Edge Selection is enabled
  • Document who has authority to trigger Data Center Exclusion
üí° Why This Is First

"Blackout and brownout protection is mostly automatic ‚Äî but only if your ZCC is current and properly configured. This is a quick validation that ensures the first two tiers of resilience are working."

Week 2 ‚Äî Define Critical Applications
Build Your Mission-Critical App List
  • Identify apps that must remain accessible during a catastrophic event
  • Separate into: internet/SaaS apps (ZIA DR allow list) and private apps (ZPA DR via PSEs)
  • Get stakeholder sign-off on the critical app list ‚Äî this is a business decision, not just IT
  • Configure the hosted DNS allow list with approved critical apps
üí° Why This Matters

"During a catastrophic event, users will only be able to access apps on this list. Defining it now ‚Äî with input from business leadership ‚Äî prevents emergency decisions made under pressure."

Week 3 ‚Äî Deploy ZPA Infrastructure
Provision Private Service Edges
  • Deploy PSE pair in primary data center or cloud (VM image or RPM on Linux)
  • Verify PSEs register with ZPA Cloud and download policies
  • Deploy and connect App Connectors for critical private applications
  • Configure DNS A records pointing to PSE public IPs
üí° Licensing Note

"One pair of Private Service Edges is included in Business Edition and above at no additional cost. Provisioning them is the single most impactful step in your resilience posture."

Week 4 ‚Äî Configure ZIA DR & Test
Complete DR Configuration and Validate
  • Register DR domain and configure DNS TXT records using Zscaler's Record Generation tool
  • Host custom PAC file externally (must be reachable when ZIA is unavailable)
  • Configure ZIA DR settings in the admin portal
  • Schedule and execute a DR drill ‚Äî validate the entire failover chain end-to-end
  • Document results and remediate any gaps
üí° The Drill Is Essential

"Configuration without testing is hope, not a plan. The drill will reveal gaps that aren't visible in documentation ‚Äî DNS propagation delays, PAC file accessibility, PSE connectivity, and user experience during failover."

Get Started

How to configure each component of Zscaler Resilience, from automatic protections to full catastrophic DR.

Step 1

Verify Client Connector Forwarding Profiles

ZCC Portal > Administration > Forwarding Profile

Ensure ZCC has both primary and secondary gateways configured for automatic blackout failover.

  • Open your forwarding profile and verify Primary Gateway and Secondary Gateway are both set
  • Primary should be closest DC to the user population; secondary should be a different DC
  • Confirm all user groups are assigned to the forwarding profile
  • Verify ZCC version across endpoints ‚Äî Windows 4.0+, macOS 3.7.1+ required for full resilience
Client Connector resilience settings ↗
Step 2

Enable Dynamic Service Edge Selection (Brownout Protection)

ZCC Portal > Administration > Forwarding Profile > Advanced

This feature lets ZCC continuously measure gateway latency and auto-switch to the best-performing path during brownouts.

  • Enable Dynamic Performance-based Service Edge Selection
  • ZCC will continuously probe configured gateways for HTTP latency
  • When the primary gateway degrades, ZCC autonomously switches to the lowest-latency alternative
  • No manual intervention required once enabled
Fail-open and brownout settings ↗
Step 3

Configure ZIA Disaster Recovery

ZIA Admin Portal > Administration > Disaster Recovery

This is the most critical pre-configuration. Without it, a catastrophic event means total loss of internet access.

  • Register a DR domain: A dedicated domain for DR DNS resolution (e.g., dr.yourdomain.com)
  • Generate DNS records: Use Zscaler's Record Generation tool to create the required TXT and A records
  • Add DNS records to your registrar: These must be in place and propagated BEFORE an event
  • Configure the hosted DNS allow list: Add your mission-critical apps (M365, Salesforce, etc.) ‚Äî only these will have direct internet access during DR
  • Host a custom PAC file: Must be reachable via HTTP/HTTPS when ZIA is unavailable ‚Äî host on infrastructure independent of Zscaler
ZIA DR configuration ↗
Step 4

Deploy ZPA Private Service Edges

ZPA Admin Portal > Administration > Service Edges

PSEs provide local ZPA brokering during a catastrophic event. One pair is included in Business Edition+ at no extra cost.

  • Provision PSEs: Deploy as VMs in your data center (VMware, Hyper-V) or cloud (AWS, Azure, GCP)
  • Register with ZPA: Verify PSEs connect to ZPA cloud and download latest policies
  • Deploy App Connectors: Ensure App Connectors for critical private apps are connected to your PSE group
  • Configure DNS A records: Point your DR domain to PSE public IPs so ZCC can reach them directly
  • Deploy in pairs: Active-active for redundancy ‚Äî both PSEs should be in the same Service Edge Group
ZPA Private Service Edge setup ↗
Step 5

Configure Tunnel Redundancy (Branch Offices)

ZIA Admin Portal > Administration > Traffic Forwarding

For branch offices using GRE or IPSec tunnels, ensure redundant tunnel configuration for failover.

  • GRE tunnels: Configure primary and secondary tunnels to different Zscaler DCs
  • IPSec tunnels: Set up redundant VPN tunnels with automatic failover on your edge device
  • Verify your router/firewall supports health-check based failover between tunnels
  • Test failover by temporarily disabling the primary tunnel

Step 6

Run a DR Drill

Configuration without testing is hope, not a plan. Validate the entire failover chain end-to-end.

  • Schedule: Coordinate with IT leadership and affected business units
  • Trigger DR mode: Follow the documented DNS-based activation procedure
  • Validate ZIA: Confirm users can access critical apps from the hosted DNS allow list
  • Validate ZPA: Confirm private app access works through PSEs
  • Measure: Time to failover, apps accessible, user experience, any gaps
  • Document: Record results, gaps discovered, and remediation steps
  • Repeat annually: DR drills should be an annual exercise at minimum
